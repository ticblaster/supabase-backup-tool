Você é um engenheiro sênior full-stack e DevOps. Gere uma ferramenta local chamada `supabase-backup-tool`, simples, segura e reutilizável, para fazer backup lógico de bancos Supabase e backup dos buckets de Storage.

## Objetivo

Criar uma aplicação/CLI local que permita apontar qualquer projeto Supabase e gerar um pacote de backup contendo:

1. Dump das roles do Postgres.
2. Dump do schema do banco.
3. Dump dos dados do banco.
4. Backup dos arquivos dos buckets do Supabase Storage.
5. Manifesto JSON com metadados do backup.
6. Guia de restauração gerado automaticamente.

A ferramenta deve servir para vários projetos Supabase, não apenas um projeto específico.

## Contexto

O Supabase no plano gratuito não fornece backup automático baixável para o usuário. Por isso, a ferramenta deve usar backup lógico via Supabase CLI/pg_dump e cópia dos arquivos dos buckets.

A ferramenta será usada inicialmente para o banco `geral-homologacao`, mas deve aceitar qualquer projeto Supabase configurado.

## Stack desejada

Use:

* Node.js com TypeScript.
* CLI interativa simples.
* Preferência por dependências maduras e poucas.
* `tsx` para execução em desenvolvimento.
* `commander` ou similar para comandos CLI.
* `dotenv` para variáveis de ambiente.
* `execa` ou `child_process` para executar comandos externos.
* `fs-extra` ou APIs nativas do Node para manipulação de arquivos.
* Estrutura limpa e fácil de manter.

Não criar interface web. A primeira versão deve ser apenas CLI.

## Pré-requisitos externos

A ferramenta pode depender de:

* Supabase CLI instalado.
* PostgreSQL client tools instalados, se necessário.
* Rclone opcional para backup avançado dos buckets.

A ferramenta deve verificar se os comandos necessários existem antes de executar:

* `supabase`
* `pg_dump`, se for usado diretamente
* `psql`, para instruções/validação de restore
* `rclone`, se o modo de storage via rclone for ativado

Se algum comando estiver ausente, mostrar erro claro com instrução de instalação.

## Requisitos funcionais

### 1. Configuração por projeto

Criar um arquivo de configuração, por exemplo:

`backup.config.json`

Com suporte a múltiplos projetos:

```json
{
  "defaultProject": "geral-homologacao",
  "backupRoot": "./backups",
  "projects": {
    "geral-homologacao": {
      "projectRef": "xxxxx",
      "dbUrlEnv": "GERAL_HOMOLOGACAO_DB_URL",
      "storage": {
        "mode": "rclone",
        "remote": "supabase-geral",
        "buckets": ["trip-receipts"]
      }
    },
    "outro-projeto": {
      "projectRef": "yyyyy",
      "dbUrlEnv": "OUTRO_PROJETO_DB_URL",
      "storage": {
        "mode": "supabase-cli",
        "buckets": ["documents", "avatars"]
      }
    }
  }
}
```

A connection string do banco não deve ficar salva no JSON. Deve ser lida de variável de ambiente.

Exemplo `.env`:

```env
GERAL_HOMOLOGACAO_DB_URL=postgresql://postgres.xxxxx:SENHA@aws-0-sa-east-1.pooler.supabase.com:5432/postgres
OUTRO_PROJETO_DB_URL=postgresql://postgres.yyyyy:SENHA@aws-0-sa-east-1.pooler.supabase.com:5432/postgres
```

Não commitar `.env`.

### 2. Comandos CLI

Implementar comandos:

```bash
npm run backup -- --project geral-homologacao
```

Executa backup completo: roles, schema, data e storage.

```bash
npm run backup:db -- --project geral-homologacao
```

Executa apenas backup do banco.

```bash
npm run backup:storage -- --project geral-homologacao
```

Executa apenas backup dos buckets.

```bash
npm run list-projects
```

Lista projetos configurados.

```bash
npm run check
```

Verifica dependências externas, arquivo de config e variáveis de ambiente.

```bash
npm run restore:print -- --project geral-homologacao --backup 20260629-1510
```

Não executa restore automaticamente. Apenas imprime um roteiro seguro de restauração com comandos sugeridos.

### 3. Estrutura do backup gerado

Para cada execução, criar uma pasta com timestamp:

```txt
backups/
  geral-homologacao/
    20260629-1510/
      db/
        roles.sql
        schema.sql
        data.sql
      storage/
        trip-receipts/
      manifest.json
      restore-notes.md
      logs/
        backup.log
```

O timestamp deve usar formato seguro para nome de pasta:

`yyyyMMdd-HHmmss`

### 4. Backup do banco

Implementar backup do banco usando Supabase CLI, preferencialmente:

```bash
supabase db dump --db-url "$DB_URL" -f roles.sql --role-only
supabase db dump --db-url "$DB_URL" -f schema.sql
supabase db dump --db-url "$DB_URL" -f data.sql --use-copy --data-only
```

Permitir exclusões opcionais no config:

```json
"db": {
  "excludeTables": [
    "storage.buckets_vectors",
    "storage.vector_indexes"
  ]
}
```

Quando houver exclusões, passar `-x tabela` no comando.

A ferramenta deve capturar stdout/stderr e salvar logs em:

```txt
logs/backup.log
```

Se qualquer etapa falhar, o processo deve parar e registrar erro.

### 5. Backup do Storage

Implementar dois modos.

#### Modo `rclone`

Usar:

```bash
rclone copy <remote>:<bucket> <backupPath>/storage/<bucket> --progress
```

Configuração:

```json
"storage": {
  "mode": "rclone",
  "remote": "supabase-geral",
  "buckets": ["trip-receipts"]
}
```

#### Modo `supabase-cli`

Usar Supabase CLI para listar/baixar quando possível. Caso a CLI não suporte download recursivo estável na versão instalada, mostrar mensagem recomendando `rclone`.

A ferramenta deve ser honesta e explícita: o backup do banco não copia os bytes dos arquivos do Storage; os buckets precisam ser copiados separadamente.

### 6. Manifesto

Gerar `manifest.json` contendo:

```json
{
  "tool": "supabase-backup-tool",
  "version": "0.1.0",
  "createdAt": "2026-06-29T18:10:00.000Z",
  "projectName": "geral-homologacao",
  "projectRef": "xxxxx",
  "backupType": "full",
  "paths": {
    "roles": "db/roles.sql",
    "schema": "db/schema.sql",
    "data": "db/data.sql",
    "storage": "storage/"
  },
  "files": [
    {
      "path": "db/schema.sql",
      "sizeBytes": 12345,
      "sha256": "..."
    }
  ],
  "storage": {
    "mode": "rclone",
    "buckets": [
      {
        "name": "trip-receipts",
        "status": "completed",
        "path": "storage/trip-receipts"
      }
    ]
  },
  "warnings": [
    "Supabase Auth, Storage metadata and managed schemas may require special validation during restore.",
    "Database dump does not include physical Storage files."
  ]
}
```

Calcular SHA-256 dos arquivos principais.

### 7. Restore notes

Gerar `restore-notes.md` com instruções claras de restauração.

Exemplo de conteúdo:

```md
# Restore notes — geral-homologacao

Backup criado em: ...

## Ordem sugerida

1. Criar novo projeto Supabase ou banco Postgres compatível.
2. Configurar variáveis de ambiente.
3. Restaurar roles.
4. Restaurar schema.
5. Restaurar dados.
6. Recriar/copiar buckets.
7. Validar RLS, Auth, Storage e fluxos principais do app.

## Comando sugerido

psql --single-transaction --variable ON_ERROR_STOP=1 \
  --file "db/roles.sql" \
  --file "db/schema.sql" \
  --command "SET session_replication_role = replica" \
  --file "db/data.sql" \
  --dbname "$TARGET_DB_URL"
```

Não implementar restore destrutivo automático na primeira versão. Apenas gerar instruções.

### 8. Segurança

Requisitos obrigatórios:

* Nunca imprimir senha completa da connection string nos logs.
* Mascarar URLs sensíveis.
* Não salvar `.env` dentro da pasta de backup.
* Não commitar backups por padrão.
* Criar `.gitignore` com:

```gitignore
.env
backups/
*.log
```

* Ao mostrar DB URL em logs, mascarar senha:

```txt
postgresql://postgres.xxxxx:****@host:5432/postgres
```
