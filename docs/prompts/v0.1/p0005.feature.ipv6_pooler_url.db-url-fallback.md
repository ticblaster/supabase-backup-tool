Atualize a ferramenta supabase-backup-tool para suportar múltiplas URLs candidatas de banco por projeto, com fallback seguro.

Contexto:
No Supabase, a conexão direta db.<project-ref>.supabase.co pode resolver apenas IPv6 em alguns ambientes. No Windows/rede local isso pode falhar em pg_dump/pg_dumpall com:
could not translate host name ... to address
ou falhas de conexão IPv6.

Queremos permitir duas URLs no .env:
- uma via Session Pooler
- outra via Direct connection

Objetivo:
A ferramenta deve tentar as URLs candidatas em ordem, escolher a primeira funcional no início do backup do banco, e usar a mesma URL escolhida para todos os dumps do banco.

Requisitos de config:
1. Manter compatibilidade com o formato antigo:
   "dbUrlEnv": "GERAL_HOMOLOGACAO_DB_URL"

2. Adicionar novo formato:
   "dbUrlEnvCandidates": [
     "GERAL_HOMOLOGACAO_DB_URL_POOLER",
     "GERAL_HOMOLOGACAO_DB_URL_DIRECT"
   ]

3. Se dbUrlEnvCandidates existir e tiver itens, usar essa lista.
4. Se não existir, usar dbUrlEnv.
5. Se ambos existirem, dbUrlEnvCandidates tem prioridade.

Exemplo de .env:

GERAL_HOMOLOGACAO_DB_URL_POOLER=postgresql://postgres.<project-ref>:SENHA@aws-0-sa-east-1.pooler.supabase.com:5432/postgres
GERAL_HOMOLOGACAO_DB_URL_DIRECT=postgresql://postgres:SENHA@db.<project-ref>.supabase.co:5432/postgres

Requisitos de seleção da URL:
1. Antes de rodar roles/schema/data, testar cada URL candidata com:
   psql --dbname "$DB_URL" --command "select 1;" --quiet --tuples-only

2. Usar a primeira URL que passar.
3. Registrar no log apenas o nome da env escolhida e a URL mascarada.
4. Nunca imprimir senha completa.
5. Se uma URL falhar por DNS, timeout ou conexão, testar a próxima.
6. Se falhar por autenticação/senha, parar imediatamente, pois fallback não deve esconder credencial errada.
7. Depois de escolher uma URL, usar a mesma URL para:
   - pg_dumpall roles.sql
   - pg_dump schema.sql
   - pg_dump data.sql
8. Não trocar URL no meio do backup.

Mensagens esperadas:
[db] Testando conexão: GERAL_HOMOLOGACAO_DB_URL_POOLER (postgresql://postgres.xxxx:****@aws-0-sa-east-1.pooler.supabase.com:5432/postgres)
[db] Conexão OK: GERAL_HOMOLOGACAO_DB_URL_POOLER
[db] Usando URL de banco: GERAL_HOMOLOGACAO_DB_URL_POOLER

Ou:
[db] Falha de conexão em GERAL_HOMOLOGACAO_DB_URL_POOLER: ...
[db] Tentando próxima URL candidata...
[db] Conexão OK: GERAL_HOMOLOGACAO_DB_URL_DIRECT

Requisitos de check:
1. npm run check deve validar que pelo menos uma URL candidata existe.
2. Se alguma env da lista estiver ausente, mostrar warning, não erro, desde que exista pelo menos uma.
3. Se nenhuma existir, erro.
4. npm run check pode testar conexão se houver flag:
   npm run check -- --project geral-homologacao --test-db-connection
5. Sem essa flag, apenas valida presença das variáveis.

Requisitos de manifest:
Adicionar ao manifest.json:
"database": {
  "engine": "pgdump",
  "selectedDbUrlEnv": "GERAL_HOMOLOGACAO_DB_URL_POOLER",
  "selectedDbUrlMasked": "postgresql://postgres.xxxx:****@host:5432/postgres",
  "candidateDbUrlEnvs": [
    "GERAL_HOMOLOGACAO_DB_URL_POOLER",
    "GERAL_HOMOLOGACAO_DB_URL_DIRECT"
  ]
}

Requisitos de testes:
Adicionar testes garantindo:
1. dbUrlEnvCandidates tem prioridade sobre dbUrlEnv.
2. fallback usa segunda URL quando a primeira falha por conexão.
3. erro de autenticação para o processo.
4. a URL escolhida é usada em roles/schema/data.
5. logs mascaram senha.
6. config antiga com dbUrlEnv continua funcionando.

Manter:
npm run typecheck
npm run build
npm run test
